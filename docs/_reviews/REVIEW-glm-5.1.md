## 0. 总体评价

**设计完备度：8.5 / 10** — 对于一个"实施前夜"的个人项目，文档体系在同类项目中属上乘。16 关卡请求生命周期、5 层候选功能架构图、跨功能横切关注点等，均表现出清晰的工程思维。

**一句话总结**：这是一份"想清楚了再动手"的设计，思路连贯、决策有据、边界清晰。但在 WSL 验证策略、daemon 化时机、数据模型细节三个方向存在可改进空间。

## 1. 亮点（做得好的地方）

### 1.1 "为什么"比"是什么"写得好

每份文档不只列出选了什么，而是清晰记录了**为什么选、为什么不选另一个**。决策日志（IMPLEMENTATION §4）是全文最有价值的部分之一——13 条决策条条有动机、有理由、有取舍。这是工程文档的黄金标准。

### 1.2 16 关卡请求生命周期设计精到

ARCHITECTURE §2 把"用户按回车到文件变更"拆成 16 道关卡，每道关卡有：
- 输入/输出边界
- 失败后果
- 实施映射（哪个步骤落地）

这种"关卡式"思考方式把复杂的 agent loop 拆成了可独立实现、可独立测试的单元。尤其 ⑨ 权限检查和 ⑫ 结果回填（错误也回传 LLM 让它自纠）的设计，显示出对 agent engineering 的深刻理解。

### 1.3 五层架构视角（BACKLOG §0）

把 7 个候选功能映射到"输入层 → 指令层 → 输出层 → 拓扑层 → 触达层"五层模型，并给出"从下到上"的实施建议。这种分层思考让功能之间的依赖关系一目了然，避免了"先做飞书还是先做 Memory"的困惑。

### 1.4 横切关注点不遗漏

ARCHITECTURE §2.5 列出了 8 个横切关注点（取消、超时、截断、循环检测、Context 降级、并发态、限流、审计），这些是实际编码中最容易遗漏但又最关键的点。尤其是：
- **大输出截断策略**（head + tail 各 25KB）— 很多人只做 head，丢了尾部 stack trace
- **循环检测用 token-set Jaccard 相似度**而不是严格 hash — 体现了对 LLM 非确定性的理解
- **⑯ 审计日志**的"不能省"论断 — 个人项目也该有

### 1.5 Scope 边界极为清晰

DESIGN §3 明确列出"做什么 / 不做什么 / 候选 / 远期 / 明确不做"五档，尤其"不做团队协作、不做云端部署、不做移动端"写得很硬。对于一个容易发散的个人项目，这种纪律性很重要。

## 2. 问题与风险（需要关注的）

### 2.1 🔴 WSL 验证是最大的定时炸弹

**现状**：步骤 1-4 都在"假设 Tauri 跑得通 WSLg"的前提下推进，步骤 5 才是首次验证。文档自己也意识到这一点（IMPLEMENTATION §2.5）：

> 此步为整个 app 的**前置硬依赖**，失败则步骤 1-4 全部回滚重新评估

**问题**：
1. Tauri 2 + WSLg 的兼容性在 2026 年 6 月仍存在不确定性。WSLg 的 Wayland 支持在某些场景下有 bug（输入法、中文渲染、HiDPI）
2. 把验证放在步骤 5 意味着前 4 步的工作可能全部浪费
3. 虽然文档建议"步骤 0 做 hello world 验证"，但这只是建议，不是路线图的硬步骤

**建议**：
- 把"WSL 验证"从建议升级为**步骤 0（硬步骤）**，在路线图中显式标注
- 验证清单至少包括：编译成功、窗口显示、中文渲染、WebView2 性能、热重载
- 同时验证 fallback 方案（VNC/X11 转发）是否可行
- 如果 Tauri + WSLg 不行，尽早决策是换技术栈（如纯 Web + 浏览器访问）还是等 Tauri 修复

### 2.2 🟡 Daemon 化时机有矛盾

**现状**：
- ARCHITECTURE §4 说"daemon 化是核心架构"
- IMPLEMENTATION §2.8 说"daemon 化占位，触发条件是飞书做不做"
- 16 关卡中 ③ 和 ⑮ 都是 daemon 化后才有的关卡

**矛盾**：
- 如果 daemon 化是"按需触发"，那步骤 1-5 的架构图跟步骤 6+ 的架构图是**完全不同的**。这会导致：
  - 代码从"Tauri 内嵌 agent"迁移到"Tauri 只是 GUI client"是重大重构
  - 前端 IPC 从"Tauri command 直接调"变成"IPC 转发到 daemon"
  - 测试策略完全不同

**建议**：
- 如果 daemon 化概率 > 50%（飞书功能看起来很有吸引力），考虑**从一开始就按 daemon 架构写**
- 即使不做飞书，daemon 架构的好处（GUI 重启不打断长任务）也值得在早期获取
- 或者：明确 MVP 阶段就是"Tauri 内嵌"，后续做 daemon 化时接受一次大重构。但要提前算清重构成本

### 2.3 🟡 数据模型缺乏具体设计

**现状**：TECH §4 说"SQLite 是唯一存储"，但 6 份文档中**没有任何 SQLite schema 定义**。

**问题**：
- session、message、tool_call、tool_result 之间的关系是核心数据模型
- 4 层 Memory 的存储位置（文件 vs SQLite）有模糊地带
- IMPLEMENTATION §3 待办里写了"SQLite schema 最终长什么样"要等做完前两步再定——但 schema 设计不应该这么晚

**建议**：
- 在 ARCHITECTURE 或 TECH 中增加一节"核心数据模型"，至少包含：
  - `projects` 表
  - `sessions` 表
  - `messages` 表（含 role、content JSON）
  - `tool_calls` 表（含 tool_use_id、name、args、result）
  - `images` / `attachments` 表
- 用 ER 图或 SQL DDL 表达，不需要最终版，但需要 v1 雏形
- Schema 跟着步骤 3（引入 SQLite）走，但设计可以提前

### 2.4 🟡 前端状态管理策略空白

**现状**：TECH §1.2 提到 Zustand vs Redux，倾向 Zustand，但没有任何前端架构设计。

**问题**：
- Chat UI 的流式 token 渲染、session 切换、tool call 状态展示——这些都是前端最复杂的部分
- 没有讨论组件结构、状态流、性能优化策略
- 对于一个"流式 + 实时"的应用，前端架构不是小事

**建议**：
- 不需要完整的前端设计文档，但在 ARCHITECTURE 中增加一节"前端架构要点"：
  - 组件树结构（至少到第二层）
  - 状态管理：哪些状态归 Zustand，哪些归 Tauri event
  - 流式渲染策略（虚拟列表？分段追加？）
  - 性能考虑（大量 token 追加时的 DOM 更新）

### 2.5 🟡 Context 压缩策略只有方向没有方案

**现状**：多处提到"context 压缩"（DESIGN §5.1 风险表、ARCHITECTURE §2.5.5 降级策略），但具体怎么压缩只说了方向（裁剪老消息 / LLM 摘要），没有落地方案。

**问题**：
- context 超限是高风险项（DESIGN §5.1 标为"高"），但唯一的缓解是"实现 context 压缩"
- 压缩策略直接影响 ⑤ 关卡（context 构造）的实现复杂度
- 什么时候触发压缩？压缩粒度是什么？压缩后的 prompt 怎么保证不丢关键信息？

**建议**：
- 在 MVP 阶段就用最简单的策略：**FIFO 裁剪**（丢最老的 tool_result 和 user/assistant 消息对）
- 记录一个简单的 token 预算表：system_prompt + memory + role + history + new_msg ≤ model_window * 0.9
- 后续再迭代到 LLM 摘要策略。但 FIFO 至少能保证不崩

### 2.6 🟢 错误处理和重试策略可以更具体

**现状**：ARCHITECTURE §2.2 第 ⑥ 关提到重试（指数退避、最多 3 次），但哪些错误重试、哪些不重试的分类不够细。

**建议**：补充一个错误分类表：

| 错误类型 | HTTP 状态码 | 策略 |
|----------|------------|------|
| 鉴权失败 | 401/403 | 立即终止，提示用户 |
| 请求过多 | 429 | 指数退避重试，读取 Retry-After |
| 服务端错误 | 500/502/503 | 重试 3 次 |
| 超时 | — | 重试 1 次 |
| 请求过大 | 400 (context overflow) | 触发压缩后重试 |
| 流中断 | SSE 断开 | 用 last_event_id 重连 |

## 6. 技术选型评估

### 6.1 rig-core vs 手写 LLM Client

文档的策略是"先手写学，后用 rig 干活"（步骤 1-2 用 reqwest，步骤 3 切 rig）。这是一个**好决策**——既保证了学习价值，又避免了长期维护样板代码。

但有一个**潜在问题**：步骤 1-2 的手写代码在步骤 3 会被**完全替换**。如果步骤 1-2 写得太"临时"，迁移成本可能比预期高。

**建议**：步骤 1-2 就把"LLM 调用"抽成 trait（`LlmClient`），这样步骤 3 只是换一个实现，不是重写。

### 6.2 SQLite 作为唯一存储：✅ 合理

个人工具、单用户、本地优先——SQLite 是完美匹配。FTS5 做搜索、sqlx 做异步、单文件零运维。

唯一要注意的是：**SQLite 不适合存超大 BLOB**（如 base64 编码的图片）。图片应该存文件系统，SQLite 只存引用——文档的 BACKLOG §1.1 已经这样设计了，但 DESIGN §5 没有提到这个约束。

### 6.3 Channel Adapter 抽象：⚠️ 可能过早

文档自己也承认了"抽象过早"的风险（ARCHITECTURE §5），但选择了"trait 只放最小接口"的缓解策略。这是合理的。

**但建议**：MVP 阶段不实现 Channel trait。直接在 Tauri command 里调用 agent core。等真正需要第二个 channel（飞书或 CLI）时再抽。过早抽象比晚抽象更危险——你不知道 trait 的实际使用模式，可能设计出不合理的接口。

## 7. 实施策略评估

### 7.1 8 步路线图：✅ 结构合理

每步有明确的目标、可交付物。步骤之间的依赖关系清晰。

### 7.2 建议：步骤 5 前移

如 2.1 所述，WSL 验证应该前移为步骤 0。不仅如此，建议把步骤 5 的**验证**部分提前，**打磨**部分保留在原位：

```
步骤 0 — WSL 硬验证（hello world + 中文渲染 + 热重载）  ← 新增
步骤 1 — 骨架 + LLM 直连
步骤 2 — Tool Calling
步骤 3 — 切 rig + 数据模型
步骤 4 — Git 集成
步骤 5 — WSL 打磨（在验证通过的基础上优化体验）
```

### 7.3 建议：增加"验收标准"

每一步的"可交付物"是定性的。建议增加具体的**验收标准**：

- 步骤 1：能发送 5 轮连续对话，SSE 不中断，流式延迟 < 200ms
- 步骤 2：agent 能成功编辑文件、执行 shell、回填结果，循环 10 轮不崩
- 步骤 3：session 切换后历史消息完整恢复，冷启动 < 2s
- 步骤 4：worktree 创建/销毁可靠，diff 视图正确渲染
- 步骤 5：Tauri 窗口在 WSLg 下显示正常，中文无乱码

## 10. 结语

Everlasting 的设计文档体系在个人项目中是罕见的完备和清晰。16 关卡请求生命周期、5 层功能架构、13 条决策日志——这些都显示出作者对"先想清楚再动手"的坚持。

最大的风险不在于设计质量，而在于**实施节奏**：WSL 验证的前移、daemon 化策略的明确、数据模型的提前设计——这三个决策如果在动手前不做，后面会反复回来咬人。

如果这份设计是一份"给自己的备忘录"（如 DESIGN §1 所述），那它已经超额完成了目标。如果它同时也要作为"实施过程中的导航图"，那上述 P0/P1 建议值得在启动步骤 1 之前补上。
