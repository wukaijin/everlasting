# spike-004: WSLg 拖拽 + 自绘顶栏按钮验证

**日期**: 2026-06-06
**状态**: **PASS**（代码 + 手测均通过，D1 可推进）
**依赖**: spike-001(WSL + Tauri 窗口能开,中文渲染 OK)
**前置决策**: `docs/spikes/003-ui-reference-prompts.md` 视觉锚点 + `research/tauri-titlebar-patterns.md` Approach B
**预估耗时**: 30-60 分钟手测 + 文档回填

## 目标

在 WSLg(Weston 合成器)下,用最小 Tauri 测试 app 验证 `data-tauri-drag-region` HTML 属性 + `decorations: false` 的实际行为。这是 D1/D4 写完整 TitleBar 组件的**前置硬依赖**:失败则必须回退到「WSL 走原生 + 其他平台自绘」feature flag(见末节"失败 → 回退方案")。

**为什么 spike 而不是直接上**:研究文件 `research/tauri-titlebar-patterns.md` §4 + §5 列出 WSLg(WebKitGTK 4.1)与 KDE Wayland / Sway / GNOME 的差异。WSLg 是 Weston 合成器,没人发过 `data-tauri-drag-region` 在 Weston 下的报告(搜过 `tauri+wsl+wslg+decorations` 无 issue)。风险:
- 拖拽静默失败(权限漏配最常见)
- 拖拽"偷焦点"导致弹出控件消失(KDE Wayland 已知问题)
- 自绘按钮 IPC 跨进程被 Rust 端 deny(权限不够)

不能等 D1/D4 写完才验证——那时失败回退成本是几百行组件代码。

## 实施内容(已完成,代码就绪)

### 改动 1:`app/src-tauri/tauri.conf.json` 窗口段

```diff
 "windows": [
   {
     "title": "Everlasting",
     "width": 960,
     "height": 720,
     "minWidth": 640,
-    "minHeight": 480
+    "minHeight": 480,
+    "titleBarStyle": "Transparent",
+    "decorations": false
   }
 ]
```

- `titleBarStyle: "Transparent"` —— Tauri 2 字段,**macOS 生效**(Transparent 让标题栏透出 webview 背景,但仍保留红绿灯),Windows/Linux 静默忽略
- `decorations: false` —— Windows/Linux 生效,删掉原生标题栏 + 边框 + 系统按钮
- 窗口尺寸(960×720)和 minSize(640×480)**未动**(那是 D1 的事)
- `trafficLightPosition` **未加**(那是 D1 的事)

> **D1 完整 plan**(参考 `research/tauri-titlebar-patterns.md` §2.1):在此基础上把窗口改成 1440×900 / min 1280×800,加 `titleBarStyle: "Overlay"`(macOS 浮层)+ `hiddenTitle: true` + `trafficLightPosition: {x: 14, y: 14}`。

### 改动 2:`app/src-tauri/capabilities/default.json`

```diff
   "permissions": [
     "core:default",
+    "core:window:allow-start-dragging",
+    "core:window:allow-minimize",
+    "core:window:allow-toggle-maximize",
+    "core:window:allow-close",
     "dialog:default"
   ]
```

4 个权限都是必须的:
- `core:window:allow-start-dragging` —— **不在 `core:window:default` 里**(已用 `target/debug/build/tauri-*/out/permissions/window/autogenerated/default.toml` 核对),漏了拖拽静默失败
- `core:window:allow-minimize` / `allow-toggle-maximize` / `allow-close` —— 同上,`core:window:default` 也没有,没加按钮点不动

> D1/D4 实施时,可视情况再加 `core:window:allow-is-maximized`(动态判断窗口最大化状态以切换按钮 icon)+ `core:window:allow-start-resize-dragging`(自绘 resize handle)。

### 改动 3:新增 `app/src/components/spike/DragTestTitleBar.vue`

~70 行 Vue 3 `<script setup>` + scoped style:
- 顶层 `<div data-tauri-drag-region>` 高 40px、Prussian tint 背景,内含一个 label(也是 drag region)
- 右侧 3 个按钮,每个加 `data-tauri-drag-region="false"` 显式 opt-out(虽然 `<button>` 默认不被 Tauri 识别为 drag region,但显式声明对未来嵌套 div 包裹的子按钮安全)
- 按钮调 `getCurrentWindow().minimize() / toggleMaximize() / close()`,try/catch 包裹并 console.error(IPC 失败时 devtools 能看到)
- 不依赖 Tailwind——纯 scoped CSS,Prussian tint `#1E2A5E` 背景是为了在 light theme 下也显眼
- 不依赖 `@tauri-apps/plugin-os`——这次不分支平台,buttons 始终显示(WSLg/Win/Linux 都是自绘;macOS 跑这个 spike 也会看到 3 按钮,与正式 D4 macOS Overlay 行为不同,记录在"已知差异"段)

## 执行步骤(给人类手测用)

### 1. 启动 dev 模式(从 `app/` 目录)

```bash
cd /usr/local/code/github/everlasting/app
pnpm tauri dev
```

启动后预期:
- 窗口出现,无 native frame(无系统标题栏、min/max/close)
- 窗口顶部 40px 高的 Prussian 蓝色条(就是我们的 drag region)
- 蓝色条左侧文字 "WSLg drag test — drag this bar (left/middle/right)"
- 蓝色条右侧 3 个按钮 `ー □ ✕`

**首次启动可能 20-30s(cargo 增量编译,spike-001 实测过)**

### 2. 验证拖拽(WSLg)

1. 鼠标在蓝色条**任意位置**按下,不要在按钮上
2. 保持按压,慢慢移动鼠标 100-200 像素
3. 期望:窗口跟随鼠标移动
4. 重复 **5 次**,每次从不同位置起拖(左、中、右)
5. 每次记录:
   - 拖动响应延迟(目测)
   - 窗口是否跟随
   - 松手后是否在预期位置

如果**完全不动**:
- 打开 DevTools(F12 或右键 → Inspect),看 Console 有无 `Failed to invoke start-dragging: permission denied` 之类
- 如果有 → 检查 `app/src-tauri/capabilities/default.json` 是否漏了 `core:window:allow-start-dragging`,**改完要重启 Tauri**(权限变更热重载不生效)
- 如果无报错且不动 → 见"失败 → 回退方案"

### 3. 验证 min/max/close(WSLg)

依次点 3 个按钮,每次点完记录:

| 按钮 | 期望 | 实际 |
|---|---|---|
| `ー`(minimize) | 窗口最小化到 dock / 任务栏 | ___ |
| `□`(toggle maximize) | 窗口铺满,再点回原尺寸 | ___ |
| `✕`(close) | 窗口关闭,`pnpm tauri dev` 进程退出 | ___ |

如果按钮点了**无反应 + console 报错**:八成是 capabilities 漏了对应权限,加完重启 Tauri。

### 4. 验证通过 → 还原 App.vue

spike 通过后,改 `app/src/App.vue`:

```diff
- import DragTestTitleBar from "./components/spike/DragTestTitleBar.vue";
+ import ChatWindow from "./components/ChatWindow.vue";

   <template>
-    <DragTestTitleBar />
+    <ChatWindow />
   </template>
```

并**保留** `app/src/components/spike/DragTestTitleBar.vue` 与 `tauri.conf.json` / `capabilities/default.json` 改动(D1 会继续用)。

### 5. 失败 → 回退方案

按代价从小到大:

#### 回退 1:用 `-webkit-app-region: drag` 替代 `data-tauri-drag-region`

理论上 WebKitGTK 不支持(`research/tauri-titlebar-patterns.md` §5.4 引 `liminal-hq/emoji-nook#5` 确认),但**实际 WSLg 下也许意外可用**(Weston 合成器实现可能有差异)。试试:

```css
.spike-titlebar {
  -webkit-app-region: drag;
}
.spike-titlebar__btn,
.spike-titlebar__controls {
  -webkit-app-region: no-drag;
}
```

#### 回退 2:WSL 走原生顶栏(其他平台自绘)

加 platform feature flag,WSLg 下完全不用 `decorations: false`,fallback 到系统原生 TitleBar:

```ts
import { platform } from "@tauri-apps/plugin-os";
const os = await platform();
const isWSL = os === "linux" && /* check WSL specific */;
if (isWSL) {
  // 不渲染 DragTestTitleBar
}
```

这要装 `@tauri-apps/plugin-os` + Rust 端 `tauri-plugin-os`,D1 文档需追加"WSL special case"段。

#### 回退 3:WSLg 临时不验证,默认假设能跑

风险高(可能 D4 实测才发现失败,白做几百行代码),不推荐。但**如果本机无法跑 WSLg**(例如换 macOS 主开发机),这是唯一选项——记录在本文件"已知未测"段,D4 上线前必须有真机验证。
